# DemoGen 产品需求文档（PRD）

> 版本：v1.0 | 2026-03-28 | 作者：DemoGen 团队

---

## 一、产品定位

### 一句话定义

**DemoGen** 是一个面向软件开发者的项目展示资产生成器——输入 GitHub 仓库链接，自动输出讲稿、PPT、一页式介绍等展示资产包。

### 核心比喻

> GitHub 项目就像淘宝上没有图片的商品——有详细的参数说明（README），但没人愿意花 20 分钟阅读文字去理解你做了什么。DemoGen 就是帮你给"商品"拍照、写描述、做详情页的工具。

### 目标用户

| 用户群体 | 场景 | 痛点 |
|---|---|---|
| **课程项目团队** | 期末答辩、课设展示 | 做了 3 周的项目，准备答辩资料又花 2 天，而且质量差 |
| **求职开发者** | 面试、作品集 | 有 GitHub 项目但不知道怎么展示给非技术面试官 |
| **独立开发者** | Product Hunt、社媒推广 | 产品做完了，宣传是另一个技能，大多数开发者不擅长 |
| **开源维护者** | 项目推广、吸引贡献者 | README 写得再好，第一次来的人也很难 30 秒内理解项目价值 |

### 我们要解决什么问题

1. **理解门槛高**：软件项目的价值藏在代码和文档里，非开发者（老师、面试官、投资人）很难快速理解
2. **展示制作慢**：准备一次高质量的项目展示（PPT + 讲稿 + 演示）需要 1-3 天
3. **展示质量差**：大多数开发者不擅长"讲故事"，做出来的 PPT 要么是代码截图堆砌，要么是空洞的技术名词

### 我们不解决什么问题

- 不是通用 PPT 生成工具（市面上已有 Gamma、Presenton 等）
- 不是 README 生成器
- 不是代码分析/审查工具
- 不替代用户理解自己的项目

## 二、竞品分析

### 直接竞品（几乎没有）

目前市面上 **没有** 一个产品专门做"GitHub 项目 → 展示资产包"这个完整流程。这是我们的机会窗口。

### 间接竞品

| 产品 | 做什么 | 和 DemoGen 的区别 |
|---|---|---|
| **Gamma.app** | 通用 AI PPT 生成 | 不理解代码项目，用户需自己描述内容 |
| **Presenton** | 开源 AI PPT 生成 | 同上，是通用工具，不针对软件项目 |
| **README2Video** | README → 宣传视频 | 只做视频一种资产，不做讲稿/PPT/one-pager |
| **ChatGPT/Claude** | 通用 AI 对话 | 能力强但无产品化流程，需用户自己组织需求和输出 |
| **PPTAgent** | Agent 框架做 PPT | 学术项目，面向研究，不是面向开发者的产品 |

### DemoGen 的差异化

1. **输入端**：直接对接 GitHub，自动获取项目信息，用户零输入成本
2. **理解层**：先理解项目再生成，不是"给我一段文字我帮你做 PPT"
3. **输出端**：一次生成完整资产包（讲稿 + PPT + one-pager），不是单一资产
4. **场景化**：针对不同展示场景（答辩/面试/推广）定制内容策略

## 三、产品流程

### 核心理念

**代码控制流程，AI 负责内容生成。**

用户不需要和 AI 聊天来推进流程。每一步由前端 UI 引导，AI 在后台按步骤生成结构化内容。聊天面板作为辅助，用于用户提问和修改。

### 用户旅程（主流程）

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 输入                                                │
│  用户粘贴 GitHub 链接（或手动输入项目描述）                      │
│  → 前端自动调用 GitHub API 获取仓库数据                         │
│  → 右侧显示加载状态                                           │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 2: 项目理解                                            │
│  前端调用 AI (generateObject) 生成结构化项目理解                │
│  → 右侧显示"项目理解卡片"（类型/功能/亮点/技术栈/风险）         │
│  → 用户可以在聊天中要求修改卡片内容                             │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 3: 场景选择                                            │
│  前端显示场景选择按钮：答辩 / 面试 / 开源推广 / 自定义          │
│  用户点击选择（或在聊天中描述自定义场景）                        │
│  可选：补充信息表单（时长限制、重点强调、观众背景）               │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 4: 策略推荐                                            │
│  前端调用 AI (generateObject) 生成展示策略                     │
│  → 右侧显示"展示策略卡片"（推荐资产/结构安排/重点）             │
│  → 用户确认或在聊天中要求调整                                  │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 5: 资产生成                                            │
│  前端按策略依次调用 AI 生成各资产：                             │
│  · 讲稿（Markdown）                                          │
│  · PPT 大纲（结构化 JSON → 可导出 PPTX）                     │
│  · One-pager（HTML 模板填充）                                │
│  → 右侧实时预览每个资产，带进度指示                             │
│  → 用户可对任意资产在聊天中要求局部修改                         │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 6: 编辑与导出                                          │
│  用户在右侧预览区编辑资产内容                                   │
│  · 讲稿：Markdown 编辑器                                     │
│  · PPT：可调整章节顺序和内容                                   │
│  · One-pager：所见即所得编辑                                   │
│  导出：讲稿 PDF / PPT 为可编辑 PPTX / One-pager 为 PDF        │
└─────────────────────────────────────────────────────────────┘
```

### 聊天面板的角色（重新定义）

聊天面板 **不是** 主流程驱动器，而是：

| 功能 | 示例 |
|---|---|
| **辅助修改** | "亮点部分换一个角度，强调性能优化" |
| **答疑解惑** | "为什么推荐做 one-pager 而不是视频？" |
| **自定义输入** | "我的项目还有一个没在 GitHub 上的功能：..." |
| **手动触发** | 没有 GitHub 链接的用户，通过描述来使用 |

### 左右面板分工

```
┌─────────────────────────┬───────────────────────────────┐
│     左侧（输入+控制）     │       右侧（预览+输出）         │
│                         │                               │
│  GitHub 链接输入框        │  项目理解卡片                  │
│  场景选择按钮            │  展示策略卡片                   │
│  补充信息表单            │  讲稿预览                      │
│  进度步骤指示器           │  PPT 大纲预览                  │
│  ─── 分隔线 ───         │  One-pager 预览               │
│  聊天对话（辅助）         │  导出按钮                      │
│                         │                               │
└─────────────────────────┴───────────────────────────────┘
```

左侧上半部分是步骤式引导（输入框 → 按钮 → 表单），下半部分是聊天。
右侧是资产预览区，随着流程推进逐步填充。

## 四、功能需求

### P0 — 核心功能（决定能不能用）

#### F1: GitHub 仓库自动解析
- 用户粘贴 GitHub 仓库 URL
- 系统自动获取：README、目录结构、package.json/依赖文件、仓库元信息（stars/language/topics）
- 解析失败时给出清晰错误提示
- **不依赖 AI**：这一步是纯 API 调用

#### F2: AI 项目理解
- 基于仓库数据，AI 生成结构化项目理解（使用 `generateObject` + Zod schema）
- 输出：项目名/类型/一句话总结/目标用户/核心功能/技术亮点/技术栈/风险
- 结果展示为右侧"项目理解卡片"
- 用户可通过聊天要求修改

#### F3: 场景选择
- 提供预设场景按钮：课程答辩、面试展示、开源推广
- 支持"自定义"选项（输入框描述）
- 可选补充信息：时长限制、重点强调方向

#### F4: 展示策略推荐
- 基于项目理解 + 用户场景，AI 生成展示策略（`generateObject`）
- 输出：推荐资产组合/观众画像/重点方向/展示结构/时间分配
- 结果展示为右侧"策略卡片"

#### F5: 讲稿生成
- 基于项目理解 + 策略，AI 生成展示讲稿（Markdown 格式）
- 讲稿结构与策略卡片的章节对应
- 右侧 Markdown 预览
- 支持导出为 PDF

#### F6: PPT 大纲生成
- 基于项目理解 + 策略，AI 生成 PPT 大纲（结构化 JSON：每页标题/要点/备注）
- 右侧卡片式预览
- 支持通过 Pandoc 导出为可编辑 PPTX

### P1 — 重要功能（决定好不好用）

#### F7: One-pager 生成
- 一页式项目介绍（HTML 渲染）
- 适合面试/社媒/README 嵌入
- 支持导出为 PDF 或图片

#### F8: 内容编辑
- 讲稿：内嵌 Markdown 编辑器，修改后实时预览
- PPT：调整章节顺序/修改要点文字
- 可通过聊天要求 AI 重写某个段落

#### F9: 聊天辅助修改
- 用户在聊天中描述修改需求
- AI 理解当前资产上下文，生成修改后的内容
- 前端替换对应部分

### P2 — 加分功能（做完 P0/P1 再考虑）

#### F10: Demo 录屏
- 对已部署 URL，用 Playwright 自动录屏
- 生成简短 GIF 或视频
- 失败时退化为截图

#### F11: 用户认证 + 历史记录
- Supabase Auth 登录
- 保存生成历史，可重新编辑

#### F12: 多项目对比
- 同时分析多个项目，生成对比展示

## 五、非功能需求

### 性能
- GitHub 解析：< 5 秒
- 项目理解生成：< 10 秒
- 单个资产生成（讲稿/PPT/one-pager）：< 15 秒
- 用流式输出保证用户感知速度

### 可靠性
- AI 输出必须符合 Zod schema，不符合则重试（最多 2 次）
- GitHub API 失败有降级方案（用户手动粘贴 README）
- 不依赖 AI 的"意愿"来推进流程

### 成本
- 单次完整流程（理解 + 策略 + 3 个资产）的 LLM 调用成本 < ¥0.5
- 使用 DeepSeek 作为默认模型可控制成本，支持切换到 GPT-4.1 提升质量

## 六、技术架构（新）

### 核心变化：从"聊天驱动"到"步骤驱动"

```
旧架构（已验证有问题）：
  用户 → 聊天 → AI 自己决定调什么工具 → 不可控

新架构：
  用户 → 前端 UI 步骤引导 → 每步调用专属 API → AI 做结构化生成 → 前端渲染
                ↕
         聊天面板（辅助修改）
```

### API 设计

| 路由 | 方法 | 功能 | AI 调用方式 |
|---|---|---|---|
| `/api/analyze` | POST | GitHub 仓库解析 | 不用 AI，纯 GitHub API |
| `/api/understand` | POST | 项目理解生成 | `generateObject` + projectUnderstandingSchema |
| `/api/strategy` | POST | 展示策略生成 | `generateObject` + displayStrategySchema |
| `/api/generate/script` | POST | 讲稿生成 | `generateText` 或 `streamText` |
| `/api/generate/ppt` | POST | PPT 大纲生成 | `generateObject` + pptOutlineSchema |
| `/api/generate/onepager` | POST | One-pager 生成 | `generateObject` + onePagerSchema |
| `/api/chat` | POST | 聊天辅助（修改、问答） | `streamText`（保留现有） |
| `/api/export/pptx` | POST | 导出 PPTX | Pandoc，不用 AI |
| `/api/export/pdf` | POST | 导出 PDF | 不用 AI |

### 前端状态机

```typescript
type Phase =
  | "input"          // 等待用户输入 GitHub 链接
  | "analyzing"      // 正在解析仓库
  | "understood"     // 项目理解完成，等待场景选择
  | "strategy"       // 展示策略已生成，等待确认
  | "generating"     // 正在生成资产
  | "review"         // 资产生成完毕，用户编辑/导出
```

每个 phase 对应左侧面板的不同 UI 状态和右侧面板的不同内容。状态机由前端代码驱动，不依赖 AI 决策。

### 技术栈（不变）

- Next.js 16 + TypeScript + App Router
- Vercel AI SDK 6（`generateObject` + `streamText`）
- assistant-ui（聊天辅助部分）
- shadcn/ui + Tailwind CSS
- Supabase（后期）
- Pandoc（PPTX 导出）

## 七、成功指标

### MVP 验收标准（学期第 10 周末）

- [ ] 输入 3 个不同类型的 GitHub 项目（Web App / CLI / API），都能稳定生成完整资产包
- [ ] 全流程无需用户在聊天中追问即可完成（步骤式 UI 引导）
- [ ] 生成的讲稿读起来像"有经验的学长写的"，不像 AI 废话
- [ ] PPT 导出为可编辑 PPTX，能用 PowerPoint/WPS 打开并修改
- [ ] 单次完整流程 < 60 秒（不含用户操作时间）

### 答辩演示策略

用 DemoGen 生成 DemoGen 自己的展示资产（meta-demo），同时准备：
1. 一个提前录好的演示视频（防翻车）
2. 现场演示一个新的 GitHub 项目输入（展示实时能力）
3. 导出的 PPTX 作为答辩 PPT 的一部分（展示真实产出）

## 八、开放问题

以下问题需要在开发过程中逐步验证：

1. **DeepSeek 的结构化输出质量够不够？** — `generateObject` 强制 schema 能提升可靠性，但内容质量取决于模型。需要用 3-5 个真实项目测试，如果不够好就切 GPT-4.1
2. **左侧面板的"步骤引导 + 聊天"混合布局怎么设计？** — 需要 UI 原型验证
3. **PPT 模板怎么做？** — Pandoc 导出的 PPTX 默认样式很丑，需要预制 2-3 个模板
4. **One-pager 的 HTML 模板** — 需要设计 1-2 个美观的模板
5. **是否需要流式输出讲稿？** — 讲稿较长，流式体验更好，但 `generateObject` 不支持流式，可能需要用 `streamText` + 后处理
